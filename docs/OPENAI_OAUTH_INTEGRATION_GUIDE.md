# `openai-oauth` 通用接入指南

## 摘要

本文件整理將 `openai-oauth` 接入任意應用時的實戰經驗，目標是讓應用可透過 ChatGPT / Codex 登入憑證，取得本機 OpenAI-compatible API 端點與可用模型列表。

這份指南刻意不依賴特定專案架構，適用於：

- 本機桌面應用
- Web 管理後台
- 單機服務
- Docker / 容器化部署

整體上要把流程拆成兩段理解：

1. **授權取得**：透過 `codex login` 取得或更新本機登入憑證。
2. **Proxy 啟動**：透過 `openai-oauth` 把這份憑證轉成可被應用調用的本地 `/v1` 相容端點。

---

## `openai-oauth` 解決什麼問題

`openai-oauth` 的角色不是模型本身，而是本地 OAuth Proxy。它會讀取你已登入的 ChatGPT / Codex 憑證，對外提供一個 OpenAI-compatible API，例如：

- Base URL：`http://127.0.0.1:10531/v1`
- 健康檢查：`http://127.0.0.1:10531/health`
- 模型列表：`http://127.0.0.1:10531/v1/models`

對應用來說，整合重點通常只有三個：

- 知道 Proxy 是否啟動成功
- 能讀到目前可用模型清單
- 把使用者選到的模型名稱保存成設定

---

## 前置條件

在接入前，先確認以下條件成立：

- 已安裝 `@openai/codex`
- 已安裝 `openai-oauth`
- 主機上有可用的登入憑證，例如 `auth.json`
- 你的應用可以安全地讀取本機憑證路徑，或至少能把該路徑傳給 Proxy
- 你理解這份憑證等同登入狀態，必須視為敏感資料保護

建議把憑證檔路徑抽成設定，例如：

- `<oauth-file>`
- `<proxy-host>`
- `<proxy-port>`

這樣本機、容器、正式環境才能用同一套程式邏輯切換。

---

## 標準接入流程

### 1. 先完成登入

優先使用一般登入流程：

```bash
codex login
```

如果環境無法自動開瀏覽器、沒有桌面工作階段，或是在遠端 / headless 機器上，再退回裝置碼流程：

```bash
codex login --device-auth
```

實務上，應用若要包這段流程，建議：

- 先嘗試 `codex login`
- 失敗或環境不支援時，再提示使用 `--device-auth`
- 把登入輸出的授權網址與授權碼解析出來，回傳給前端或 UI 顯示

### 2. 啟動本地 Proxy

```bash
openai-oauth --host <proxy-host> --port <proxy-port> --oauth-file <oauth-file>
```

常見本機開發值：

```bash
openai-oauth --host 127.0.0.1 --port 10531 --oauth-file <oauth-file>
```

### 3. 驗證 Proxy 狀態

至少檢查這兩個端點：

- `/health`：確認 Proxy 已就緒
- `/v1/models`：確認憑證有效，且能成功列出模型

建議驗證順序：

1. 先打 `/health`
2. 再打 `/v1/models`
3. 若 `/health` 正常但 `/v1/models` 失敗，優先懷疑憑證失效或登入流程未完成

### 4. 在應用中讀取模型清單

不要在前端或後端硬編碼模型名稱。應以 `/v1/models` 為唯一來源，動態轉成：

- `label`
- `value`

應用端最小可用設計通常是：

- 顯示 Proxy 狀態
- 顯示可用模型下拉選單
- 保存使用者選中的模型名稱

## 應用側整合建議

### 抽象成四種狀態

不論是 Web、桌面端或 CLI 包裝層，都建議把整合狀態抽成四類：

- `未登入`：找不到憑證或憑證明顯無效
- `等待授權`：登入流程已啟動，但尚未完成瀏覽器授權
- `代理運行中`：Proxy 已啟動，且可讀取模型
- `代理異常`：Proxy 啟動失敗、模型列表讀取失敗、或回傳 `401`

### UI / 設定頁的最小交付

如果你的應用有設定頁，建議至少提供：

- 一個「開始登入」入口
- 一個 Proxy 狀態顯示
- 一個模型下拉選單
- 一個 Base URL 顯示或預設值

其中模型名稱不應是自由輸入，而應直接綁定 `/v1/models` 的回傳結果。

### 錯誤訊息要可操作

不要只顯示「登入失敗」或「代理失敗」。比較有用的訊息應該能直接指向下一步，例如：

- 找不到 `auth.json`
- Proxy 連不上
- 憑證已失效，請重新登入
- 模型列表為空，請檢查登入是否完成
- Port 已被占用

## Docker / 容器整合經驗

這是最容易踩坑的部分。

### 1. 不要期待容器內命令能自動打開宿主瀏覽器

在容器內執行 `codex login` 時，即使它印出瀏覽器授權網址，也通常無法直接打開宿主機瀏覽器。這不是登入邏輯壞掉，而是容器沒有桌面環境。

正確做法通常是：

- 後端解析登入輸出中的 `authUrl`
- 把 `authUrl` 回傳給前端或 UI
- 由宿主環境的瀏覽器去開啟該網址

### 2. `localhost` 回呼不是同一個 `localhost`

很多登入流程會啟動本地回呼，例如：

`http://localhost:1455/auth/callback`

但要注意：

- 瀏覽器中的 `localhost` 是宿主機
- 容器內程式監聽的 `localhost` 是容器自己

如果登入程式跑在容器裡，而瀏覽器開在宿主機，回呼通常不會自動送回容器內的服務，除非你額外做轉發。

### 3. 宿主與容器之間要明確做 1455 轉發

若你的登入流程依賴本地回呼，常見做法是：

- 將宿主 `127.0.0.1:1455` 映射進容器
- 或在容器內做一層 bridge，把容器實際 IP 的 `1455` 轉給 `127.0.0.1:1455`

這樣瀏覽器回到宿主的 `http://localhost:1455/...` 後，才能最終送進容器裡正在等待回呼的登入程序。

### 4. 何時用 `127.0.0.1`，何時用 `0.0.0.0`

- 單機、本機 UI 與 Proxy 同機：優先綁 `127.0.0.1`
- 需要跨容器、跨服務或由宿主外其他程序訪問：再考慮 `0.0.0.0`

預設原則很簡單：**能綁 loopback 就不要暴露到所有網卡**。

## 常見坑與排錯

### 找不到 `auth.json`

症狀：

- Proxy 啟動失敗
- `/v1/models` 無法讀取
- 應用顯示未登入

檢查方向：

- `codex login` 是否真的成功完成
- `<oauth-file>` 路徑是否正確
- 容器是否真的掛載到正確的憑證目錄
- 執行 Proxy 的使用者是否有權限讀檔

### `/health` 正常，但 `/v1/models` 失敗

症狀：

- Proxy 看似已啟動
- 但模型清單讀不到

通常代表：

- 憑證已失效
- 登入尚未完成
- OAuth token 需要重新刷新

處理順序：

1. 重新登入
2. 重啟 Proxy
3. 再驗證 `/v1/models`

### 出現 `401` 或 `invalid_token`

通常不是應用程式本身邏輯錯，而是憑證已過期或本地憑證狀態不一致。

建議動作：

- 重新執行 `codex login`
- 確認 Proxy 讀取的真的是最新憑證
- 如果有多份憑證檔，避免讀到舊路徑

### Device Auth 卡住

症狀：

- 命令列停在等待授權
- 使用者不知道該去哪裡完成登入

建議：

- 把授權網址直接顯示在 UI
- 把一次性授權碼直接顯示在 UI
- 即使你有自動開頁，也要保留手動開啟連結的能力

### 容器內 `codex login` 無法完成回呼

這通常不是 `openai-oauth` 本身的問題，而是登入回呼回到了宿主 `localhost`，沒有轉發到容器裡。

優先檢查：

- `1455` 是否有對應的 host/container 映射
- 是否有其他程式先占用這個埠
- bridge 是否綁錯位址，導致和登入程式搶同一個 `127.0.0.1:1455`

### Port 被占用

常見於：

- Proxy 的 `<proxy-port>` 已被別的服務使用
- 登入回呼埠 `1455` 已被其他進程占用

建議：

- 讓應用把埠占用明確寫進錯誤訊息
- 若登入與 bridge 併用，避免兩者都綁同一個 loopback 位址

## 安全建議

`openai-oauth` 整合的核心風險不在模型調用，而在憑證與本地代理暴露。

最低限度要做到：

- `auth.json` 不提交到版本控制
- 不把憑證掛到公開可下載的目錄
- Proxy 預設只監聽 `127.0.0.1`
- 若必須跨容器或跨主機，額外加上網路邊界與權限控制
- 不在日誌中輸出完整憑證內容
- 對授權網址、授權碼、模型列表的顯示做最小暴露

如果要讓團隊多人共用，應先明確界定：

- 誰可以讀憑證
- 誰可以啟動 Proxy
- 誰可以從應用端直接使用該 Proxy

---

## 最小可用整合清單

若你只想快速接起來，至少完成以下項目：

1. 可執行 `codex login`
2. 可執行 `openai-oauth --host <proxy-host> --port <proxy-port> --oauth-file <oauth-file>`
3. 應用能檢查 `/health`
4. 應用能讀取 `/v1/models`
5. UI 或設定檔能保存選定模型
6. 錯誤訊息能區分「未登入 / 代理未啟動 / 憑證失效 / 埠占用」

---

## 部署前檢查清單

上線前建議逐項確認：

- 一般登入流程可用
- `--device-auth` fallback 可用
- Proxy 重啟後仍能正常讀取憑證
- `/health` 與 `/v1/models` 在目標環境都可通
- 模型清單不是硬編碼，而是動態讀取
- 容器場景下，登入回呼鏈路已驗證
- Proxy 沒有不必要地暴露到公網

---

## 實戰經驗抽象

如果要把整體經驗濃縮成幾條原則，可以記成：

- 把「登入」和「代理」視為兩套流程，不要混成一個黑盒
- 把 `authUrl`、`authCode`、Proxy 狀態、模型列表都顯式暴露給 UI 或日誌
- 模型清單永遠以 `/v1/models` 為準，不要前端硬編碼
- 容器整合時，優先處理 `localhost` 回呼路徑，而不是一開始就懷疑 OAuth 本身
- 能綁 `127.0.0.1` 就不要綁 `0.0.0.0`

這樣即使換一個專案、換一個前端框架、換成不同部署方式，整體接入思路仍然成立。
